# Deployment Plan — onesearch.blog

## Overview

Host OneSearch on a **Hetzner VPS** — a reliable, developer-favourite cloud provider that costs €4.49/month for 4GB RAM. No laptop needs to be on, no Cloudflare tunnel, and your domain `onesearch.blog` points directly to the server.

```
GitHub push → GitHub Actions (tests) → SSH → Hetzner VPS
                                                   │
                                         ┌─────────┴──────────┐
                                         │  Docker Compose      │
                                         │  ├── app (Flask)     │
                                         │  ├── nginx (SSL)     │
                                         │  └── certbot         │
                                         └─────────────────────┘
                                                   │
                                         https://onesearch.blog
```

### Why this stack?
- **Hetzner** — cheapest reliable VPS with enough RAM (4GB) for your ML model. DigitalOcean costs 2-3x more for half the RAM.
- **Docker Compose** — you already have it, keeps the app portable and easy to redeploy.
- **Nginx** — sits in front of Flask to handle SSL termination and serve as a proper reverse proxy. Flask's built-in server isn't meant for production traffic.
- **Let's Encrypt / Certbot** — free SSL certificates, auto-renewable. Required for HTTPS on your domain.
- **GitHub Actions SSH deploy** — on every push to `main`, Actions SSHs into the server and pulls + rebuilds. No manual deploys needed.

## Step 1 — Create a Hetzner Server

**Why:** You need an always-on machine with a public IP. Hetzner gives you 4GB RAM for €4.49/month — enough to run Flask, load the ML model (~1.5GB RAM), and handle background jobs without running out of memory.

1. Sign up at [hetzner.com/cloud](https://hetzner.com/cloud) (no bureaucracy, card billed monthly)
2. Create a new **Project** (e.g., `onesearch`)
3. Click **Add Server**:
   - **Location**: pick closest to your users — Nuremberg/Helsinki for EU, Ashburn for US
   - **Image**: Ubuntu 22.04
   - **Type**: **CX22** — 2 vCPU, 4GB RAM, 40GB SSD, €4.49/mo
   - **SSH Key**: paste your local public key (`~/.ssh/id_rsa.pub`) so you can log in without a password
4. Click **Create** — you get a public IP in about 30 seconds

## Step 2 — Point onesearch.blog DNS to the Server

**Why:** DNS tells the internet where `onesearch.blog` lives. Without this, visitors can't reach your server by domain name — only by raw IP. TTL of 300 seconds means changes propagate in ~5 minutes.

In Namecheap → Advanced DNS for `onesearch.blog`:

| Type | Host | Value              | TTL |
|------|------|--------------------|-----|
| A    | @    | `<hetzner-ip>`     | 300 |
| A    | www  | `<hetzner-ip>`     | 300 |

Verify propagation:
```bash
dig onesearch.blog +short
# should return your Hetzner IP
```

## Step 3 — Set Up the Server

**Why:** The server is a blank Ubuntu machine. You need Docker to run your containers, and the repo to have the app code.

SSH in:
```bash
ssh root@<hetzner-ip>
```

Install Docker (official one-liner, installs Docker + Compose plugin):
```bash
curl -fsSL https://get.docker.com | sh
```

Clone your repo:
```bash
git clone https://github.com/<your-org>/onesearch.git /app
cd /app
```

> **No `.env` file needed here.** Secrets are stored in GitHub and written to the VM automatically on every deploy (see Step 9). The `.env` file is generated by GitHub Actions — you never manage it manually on the server.

## Step 4 — Update docker-compose.yml

**Why:** The current compose file binds Flask directly to the host port and uses Cloudflare tunnel. In production you want Nginx in front of Flask (handles SSL, compression, proper HTTP headers) and no tunnel dependency. Certbot handles SSL certificate issuance and renewal.

Replace the contents of `docker-compose.yml`:

```yaml
version: "3.9"
services:
  app:
    build: .
    expose:
      - "8000"          # only reachable inside Docker network, not from outside
    env_file:
      - .env
    read_only: true
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
    pids_limit: 100
    mem_limit: 512m
    cpus: "0.5"
    volumes:
      - ./logs:/app/logs
      - ./tmp:/app/tmp
      - ./data:/app/data
      - ./static:/app/static
    networks:
      - appnet
    container_name: onesearch_app
    restart: unless-stopped

  nginx:
    image: nginx:alpine
    container_name: onesearch_nginx
    ports:
      - "80:80"       # HTTP (redirects to HTTPS)
      - "443:443"     # HTTPS
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
      - certbot-certs:/etc/letsencrypt
      - certbot-www:/var/www/certbot
    depends_on:
      - app
    networks:
      - appnet
    restart: unless-stopped

  certbot:
    image: certbot/certbot
    container_name: onesearch_certbot
    volumes:
      - certbot-certs:/etc/letsencrypt
      - certbot-www:/var/www/certbot

volumes:
  certbot-certs:   # stores SSL certificates, persisted across restarts
  certbot-www:     # used by certbot for domain verification challenge files

networks:
  appnet:
```

## Step 5 — Create nginx.conf (Bootstrap — HTTP only)

**Why:** Before you can get an SSL certificate, Certbot needs to verify you own the domain by placing a file at `onesearch.blog/.well-known/acme-challenge/`. Nginx must be running on port 80 to serve that file. So you start with an HTTP-only config, get the cert, then switch to HTTPS.

Create `nginx.conf` in the repo root:

```nginx
server {
    listen 80;
    server_name onesearch.blog www.onesearch.blog;

    # Certbot uses this path to verify domain ownership
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # Forward all other traffic to Flask
    location / {
        proxy_pass http://app:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

## Step 6 — Get SSL Certificate

**Why:** HTTPS is required for a production site — browsers show "Not Secure" warnings without it, and some email clients reject links from HTTP sites. Let's Encrypt issues free certificates valid for 90 days. Certbot handles issuance and renewal automatically.

Start Nginx and the app first:
```bash
docker compose up -d app nginx
```

Run Certbot to issue the certificate:
```bash
docker compose run --rm certbot certonly \
  --webroot -w /var/www/certbot \
  -d onesearch.blog -d www.onesearch.blog \
  --email your@email.com --agree-tos --no-eff-email
```

Now update `nginx.conf` to enable HTTPS and redirect HTTP:

```nginx
# Redirect all HTTP to HTTPS
server {
    listen 80;
    server_name onesearch.blog www.onesearch.blog;

    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location / {
        return 301 https://onesearch.blog$request_uri;
    }
}

# HTTPS server
server {
    listen 443 ssl;
    server_name onesearch.blog www.onesearch.blog;

    ssl_certificate /etc/letsencrypt/live/onesearch.blog/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/onesearch.blog/privkey.pem;

    location / {
        proxy_pass http://app:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto https;
    }
}
```

Reload Nginx to pick up the new config:
```bash
docker compose restart nginx
```

## Step 7 — Set Up SSL Auto-Renewal

**Why:** Let's Encrypt certificates expire after 90 days. Without auto-renewal, your site will show SSL errors and stop working. A daily cron job checks if renewal is needed (it only renews when <30 days remain) and reloads Nginx to pick up the new cert.

```bash
crontab -e
```

Add:
```
0 3 * * * docker compose -f /app/docker-compose.yml run --rm certbot renew --quiet && docker compose -f /app/docker-compose.yml restart nginx
```

## Step 8 — Background Jobs (Cron)

**Why:** Your pipeline (`scrape_pubs.py` → `notify.py` → `send_notifications.py`) needs to run on a schedule. Running them inside the existing `app` container via `docker exec` is the simplest approach — no extra containers, no separate scheduler service.

```bash
crontab -e
```

Add:
```
# Fetch new blog posts every 2 hours
0 */2 * * *  docker exec onesearch_app python scrape_pubs.py

# Build and send daily digest at 8am
0 8 * * *    docker exec onesearch_app python notify.py
5 8 * * *    docker exec onesearch_app python send_notifications.py

# Retrain ML classifier every Sunday at 2am
0 2 * * 0    docker exec onesearch_app python training.py
```

## Step 9 — Update GitHub Actions

**Why:** The existing workflow deploys to Azure AKS, which you're no longer using. The new workflow builds and tests the image, then SSHs into Hetzner to pull code, write the `.env` from GitHub Secrets, and rebuild the container. Secrets are managed in one place (GitHub) — no `.env` file to maintain manually on the server. Tests run before deploy — if pytest fails, the deploy is skipped.

Add these in GitHub → Settings → Secrets and Variables → Actions:

**Secrets** (sensitive values, encrypted):
| Secret | Value |
|--------|-------|
| `VPS_HOST` | Your Hetzner server IP |
| `VPS_SSH_KEY` | Contents of `~/.ssh/id_rsa` (your private key) |
| `POSTS_SECRET_KEY` | A long random string for admin endpoints |
| `SMTP_USERNAME` | Your Zoho email address |
| `SMTP_PASSWORD` | Your Zoho app password |
| `SECRET_KEY` | Flask secret key (random string) |

**Variables** (non-sensitive config):
| Variable | Value |
|----------|-------|
| `FLASK_ENV` | `production` |
| `DB_PATH` | `data/onesearch_prod.db` |
| `MODEL_PATH` | `data/trained_classifier.pkl` |

Replace `.github/workflows/ci-cd.yml`:

```yaml
name: CI/CD to Hetzner

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Build Docker image
        run: docker build -t onesearch:test .

      - name: Run tests
        run: docker run --rm onesearch:test pytest

  deploy:
    needs: build-and-test
    if: github.ref == 'refs/heads/main'   # only deploy on main, not PRs
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Hetzner
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VPS_HOST }}
          username: root
          key: ${{ secrets.VPS_SSH_KEY }}
          port: 443   # SSH runs on 443 to work through corporate firewalls
          envs: FLASK_ENV,DB_PATH,MODEL_PATH,POSTS_SECRET_KEY,SMTP_USERNAME,SMTP_PASSWORD,SECRET_KEY
          script: |
            cd /app
            git pull origin main

            # Write .env from GitHub Secrets — no manual .env management on server
            cat > .env << EOF
            FLASK_ENV=$FLASK_ENV
            DB_PATH=$DB_PATH
            MODEL_PATH=$MODEL_PATH
            POSTS_SECRET_KEY=$POSTS_SECRET_KEY
            SMTP_USERNAME=$SMTP_USERNAME
            SMTP_PASSWORD=$SMTP_PASSWORD
            SECRET_KEY=$SECRET_KEY
            EOF

            docker compose up -d --build
        env:
          FLASK_ENV: ${{ vars.FLASK_ENV }}
          DB_PATH: ${{ vars.DB_PATH }}
          MODEL_PATH: ${{ vars.MODEL_PATH }}
          POSTS_SECRET_KEY: ${{ secrets.POSTS_SECRET_KEY }}
          SMTP_USERNAME: ${{ secrets.SMTP_USERNAME }}
          SMTP_PASSWORD: ${{ secrets.SMTP_PASSWORD }}
          SECRET_KEY: ${{ secrets.SECRET_KEY }}
```

## What's Not Covered

- **Monitoring** — Uptime Robot (free) can ping `onesearch.blog` every 5 min and email you if it goes down.
- **Backups** — Hetzner offers automated backups for +20% of server cost (€~0.90/mo). Worth it to protect your SQLite DB.
- **Scaling** — If you outgrow SQLite or need more than one server, the next step would be moving to Postgres. That's not needed now.
